BPMN flow control — two related additions

Both tickets touch the same subsystem: the BPA-backend Camunda 7 BPMN schema generator and the notion of branching control in service workflows. They extend — not replace — today's deterministic model. Both are designed to stay fully backwards compatible: services that don't opt in produce byte-identical BPMN.

  • TOBE-17769 adds a targeted return loop — send a task to one role and wait for it, without restarting the downstream flow.
  • TOBE-17771 adds dead-ending branches — let one parallel branch terminate cleanly without killing the whole process at the join.

Conceptually: 17769 is about returning (backwards/inwards control), 17771 is about abandoning (forwards branch termination). Different primitives, same layer of code.

Resubmit to role only

Story · Priority High · In DevelopmentComponent: CamundaAssignee: Erki Kriks

Today

When a reviewer declines and sends back, the BPMN generator wires the task back into the main sequence flow at that role. Execution resumes forward from that point — everyone downstream runs again. A FileDeclineStatus has a RoleStatusDestination that just picks the target role; there is no notion of how the control should come back.

What's missing

A way to say: "loop this one role only, then return to me" — a boomerang instead of a restart.

Minimum-change concept

  1. Model — add one optional field on RoleStatusDestination (or on the send-back action): resubmissionMode = DETERMINISTIC (default) | ROLE_ONLY. Null or absent preserves current semantics.
  2. Schema generator (ExclusiveBuilder) — when ROLE_ONLY:
    • Don't re-enter the main sequence at the target role.
    • Emit a Camunda Call Activity (or bounded sub-process) scoped to that one role's user task, with the originating role's task parked on a message/signal event.
    • On sub-flow completion, signal back to the originator — the process resumes exactly where it paused.
  3. Form UI (SendBackModalWindowGenerator) — if the service author enabled ROLE_ONLY on that transition, render a radio: "Full re-review" vs "Just fix and return". Otherwise no UI change.
  4. Runtime (DS-Backend) — no logic change; Camunda handles sub-process lifecycle natively. Only the return-signal callback is new.
  5. Edge case (from the AC) — if the "role-only" target declines or never responds, the sub-process terminates with a signal that the originator interprets as a forced deterministic fallback (revert to today's behavior rather than deadlock). Configurable, or hardcoded per safety.
Why minimum-change: one enum field, one branch in one builder, reuses Camunda's native call-activity and message-event primitives. BPMN output for every currently deployed service is byte-for-byte unchanged.

Dead branches in part B

Story · Priority Highest · To DoReference: Jamaica SEZ service

Today

Part B (post-submission back-office) can fork into parallel roles via an inclusive gateway, then join. For the join to fire, each active branch must emit a FileValidatedStatus. If a role instead issues a FileRejectStatus, the whole process terminates — there is no concept of "this objection kills its branch but the case continues" or "this branch can be skipped without starving the join".

Jamaica pattern

The SEZ-Developer service (v70) has 8 parallel registrations: three "No objection" steps (JCA, MOFPS, TAJ) and four screenings (BPSS, CAS, LSU, TSI). A "No objection" role may legitimately not weigh in; one screening may reject while others approve — yet the case must still reach the coordinator. Branches that go nowhere must not block the join.

Minimum-change concept

  1. Model — add one optional flag on Role (or on the fork entry in the graph): branchTerminationMode = BLOCKING (default = today) | DEAD_END_ON_REJECT | OPTIONAL. Unset = today's behavior.
  2. Schema generator (JoinSequenceBuilder + ExclusiveBuilder) — when a branch is marked dead-end-capable:
    • On FileRejectStatus, route that branch to a terminate-branch end event scoped to the fork's sub-process, not the global terminate. Camunda's inclusive join already tolerates tokens that never arrive when not activated; we leverage that by producing a token-consuming terminate event per branch.
    • Extend the DMN decision table at the join with a "branch dead" input so the downstream decision explicitly knows which branches didn't vote.
  3. Runtime (DS-Backend) — no change. Camunda's inclusive-join semantics already handle arrived-vs-didn't-arrive tokens.
  4. UI — service author sees a new checkbox on the role: "Allow this branch to dead-end on reject". No citizen-facing change.
Why minimum-change: exploits Camunda's existing inclusive-gateway + terminate-end-event primitives. Changes confined to two builders plus one DMN column. Services that don't set the flag produce identical BPMN as today.
