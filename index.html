Two related additions to the workflow engine

Conceptual analysis of TOBE-17769 and TOBE-17771 — plain-English approach, minimum code change, full backwards compatibility.

Both tickets touch the same subsystem: the BPA-backend Camunda 7 BPMN schema generator. They extend — not replace — today's deterministic model. Services that don't opt in produce byte-identical BPMN.

  • TOBE-17769 — a targeted return loop: send a task to one role and wait for it, without restarting the downstream flow.
  • TOBE-17771 — dead-ending branches: let one parallel branch terminate cleanly without killing the whole process at the join.

17769 is about returning (backwards / inwards control). 17771 is about abandoning (forwards branch termination). Different primitives, same layer of code.

Resubmit to role only TOBE-17769

Story · Priority High · In DevelopmentComponent: CamundaAssignee: Erki Kriks

The story, in plain English

Imagine a business license application. The applicant submits. Then four officers review it in order: Officer A checks identity, Officer B checks the company documents, Officer C checks tax status, Officer D signs off.

Today, if Officer C spots a small problem — say, a mistyped tax ID — and sends the file back to the applicant, the applicant fixes it, and then the whole review restarts: Officer A checks identity again, Officer B re-reviews the company docs, and only then does C look again. Everyone has to re-do their part, even though only one tiny field changed.

What we want to add: Officer C should be able to say "just send it to the applicant to fix this one thing, then bring it straight back to me." The applicant fixes it, and it lands directly back on Officer C's desk — A, B, and D don't see it again. It's a targeted loop instead of a full restart.

Both options must exist side-by-side: sometimes you do want a full re-review (say, a new document was added that changes everything); sometimes you just want the one correction. The officer chooses which mode when they send back.

At a glance Add an optional "resubmission mode" setting on the send-back action. When set to "role only," the workflow pauses the sender's task, sends the file to just that one role, and resumes the paused task when the role is done — instead of restarting the main flow.

Technical breakdown (for developers)

Today

When a reviewer declines and sends back, the BPMN generator wires the task back into the main sequence flow at that role. Execution resumes forward from that point — everyone downstream runs again. A FileDeclineStatus has a RoleStatusDestination that just picks the target role; there is no notion of how control should come back.

What's missing

A way to say: "loop this one role only, then return to me" — a boomerang instead of a restart.

Minimum-change concept

  1. Model — add one optional field on RoleStatusDestination (or on the send-back action): resubmissionMode = DETERMINISTIC (default) | ROLE_ONLY. Null or absent preserves current semantics.
  2. Schema generator (ExclusiveBuilder) — when ROLE_ONLY:
    • Don't re-enter the main sequence at the target role.
    • Emit a Camunda call activity (or bounded sub-process) scoped to that one role's user task, with the originating role's task parked on a message/signal event.
    • On sub-flow completion, signal back to the originator — the process resumes exactly where it paused.
  3. Form UI (SendBackModalWindowGenerator) — if the service author enabled ROLE_ONLY on that transition, render a radio: "Full re-review" vs "Just fix and return". Otherwise no UI change.
  4. Runtime (DS-Backend) — no business logic change; Camunda handles sub-process lifecycle natively. Minor: the task view may need to recognize the "returned from role-only sub-flow" state so the originator's UI shows the right label.
  5. Edge case (from the AC) — if the "role-only" target declines or never responds, the sub-process terminates with a signal that the originator interprets as a forced deterministic fallback (revert to today's behavior rather than deadlock).
Why minimum-change: one enum field, one branch in one builder, reuses Camunda's native call-activity and message-event primitives. BPMN output for every currently deployed service is byte-for-byte unchanged.

Dead branches in part B TOBE-17771

Story · Priority Highest · To DoReference: Jamaica SEZ service

The story, in plain English

Think of Jamaica's Special Economic Zone application. After the applicant submits, eight different agencies review the file at the same time, in parallel: tax authority, customs, legal, planning, and so on. Each agency either gives its green light or raises a flag. Once all eight have spoken, a coordinator takes the final decision.

Today, if any one of those eight agencies rejects, the entire case dies immediately. The other seven agencies — some of whom may have already approved — never get a chance to finish, and the coordinator never even sees the case. A single "no" blows up everything.

What we want to add: allow a rejection from one agency to end just that agency's branch — not the whole case. The other agencies finish their work normally. The coordinator sees the full picture: "five approved, two approved with comments, one rejected" — and decides.

The word "dead branch" means: this branch is done, it contributed nothing, and we just move on without it. The case stays alive.

Also: some agencies (like the three "No objection" bodies in the Jamaica flow) are optional — they may not weigh in at all, and the case should proceed without waiting for them forever. Same mechanism.

At a glance Add an optional flag on each parallel role saying "this branch is allowed to dead-end." When set, a rejection from that role ends its branch cleanly without killing the whole case — the coordinator still gets the final decision with whatever input arrived.

Technical breakdown (for developers)

Today

Part B (post-submission back-office) can fork into parallel roles via an inclusive gateway, then join. For the join to fire, each active branch must emit a FileValidatedStatus. If a role instead issues a FileRejectStatus, the whole process terminates — there is no concept of "this objection kills its branch but the case continues" or "this branch can be skipped without starving the join".

Jamaica pattern

The SEZ-Developer service (v70) has 8 parallel registrations: three "No objection" steps (JCA, MOFPS, TAJ) and four screenings (BPSS, CAS, LSU, TSI). A "No objection" role may legitimately not weigh in; one screening may reject while others approve — yet the case must still reach the coordinator. Branches that go nowhere must not block the join.

Minimum-change concept

  1. Model — add one optional flag on Role (or on the fork entry in the graph): branchTerminationMode = BLOCKING (default = today) | DEAD_END_ON_REJECT | OPTIONAL. Unset = today's behavior.
  2. Schema generator (JoinSequenceBuilder + ExclusiveBuilder) — when a branch is marked dead-end-capable:
    • On FileRejectStatus, route that branch to a terminate-branch end event scoped to the fork's sub-process, not the global terminate. Camunda's inclusive join already tolerates tokens that never arrive; we leverage that by producing a token-consuming terminate event per branch.
    • Extend the DMN decision table at the join with a "branch dead" input so the downstream decision explicitly knows which branches didn't vote.
  3. Runtime (DS-Backend) — no change. Camunda's inclusive-join semantics already handle arrived-vs-didn't-arrive tokens.
  4. UI — service author sees a new checkbox on the role: "Allow this branch to dead-end on reject". No citizen-facing change.
Why minimum-change: exploits Camunda's existing inclusive-gateway + terminate-end-event primitives. Changes confined to two builders plus one DMN column. Services that don't set the flag produce identical BPMN as today.

Where the work actually lives

Erki's framing is correct: this is a BPA-backend change, not a Camunda change. Here's the precise breakdown so there's no ambiguity during review.

BPA-backend — the schema generator (primary surface)

BPMN XML is generated server-side in Java by:

  • ProcessBuilder.generateBPMN() — orchestrator
  • ExclusiveBuilder — emits decline / reject routing (touched by both tickets)
  • JoinSequenceBuilder — emits inclusive fork-join structures (touched by TOBE-17771)
  • SendBackModalWindowGenerator — emits the Form.io modal shown to reviewers (touched by TOBE-17769)

Model additions: RoleStatusDestination.resubmissionMode (17769), Role.branchTerminationMode (17771). Both nullable, both default to today's behavior.

DS-Backend — runtime glue

  • TOBE-17769 — minor: the task view that handles send-back completion may need to recognize the "returned from role-only sub-flow" case so the UI label is correct. No new Camunda API calls.
  • TOBE-17771 — zero. Camunda handles inclusive-join with missing tokens natively.

Camunda — the engine

  • No code written. No fork of Camunda. No custom delegates.
  • We only emit BPMN elements Camunda 7 already supports: call activity, message/signal intermediate events, terminate end event, inclusive gateway, DMN business rule task.
Caveat — the testing matters. We don't modify Camunda, but we still have to verify at runtime that:
  • TOBE-17769 — the return path (sub-flow signal → parent resumes parked task) works end-to-end and process variables survive the round-trip.
  • TOBE-17771 — the inclusive-join reliably fires when one branch ends via a scoped terminate event while others approve.
Both are integration-test territory, not Camunda code.
Bottom line for Erki's message: "BPMN schemas are generated in BPA side. Camunda is just runtime" — correct. The change is in the generator; Camunda is exercised through standard primitives and validated via integration tests.
